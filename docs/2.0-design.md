# 2.0.0 设计草案

本文记录 QuarkfanTools 2.0.0 的实现方向。2.0.0 的核心变化是从“Bot 授权 Skills 后由 Agent 自主选择”扩展为“Bot 挂载一组可治理能力，用户、Agent、命令和定时任务都能在同一权限边界内调用”。

## 1. 目标

- 在每个 Bot 维度支持定时任务，由应用本身负责调度、持久化、暂停、恢复和执行记录。
- 支持 MCP 服务配置，并允许 Bot 在授权范围内使用 MCP 工具。
- 引入套件概念，用于面向角色或行业组合 Skills、MCP、工作流和说明。
- 引入 `/xxxx` 命令机制，命令可配置为调用 Skill、套件或后续扩展工具。
- 支持跟随系统、浅色和深色三种主题。

## 2. 非目标

- 2.0.0 首版不实现任意远端调度服务；定时任务只在本机应用运行时触发。
- 2.0.0 首版不开放任意 shell 命令作为命令目标，避免绕过 Skill/MCP/套件的授权模型。
- 2.0.0 首版命令映射支持 Skill、MCP、套件、Workflow 和自定义应用。
- 套件不是新的安全边界。最终安全边界仍是 Bot 授权、sandbox、目录隔离和 MCP 服务配置。

## 3. Bot 可治理能力模型

2.0.0 的底层抽象是 Bot Capability Governance。所有能被 Bot 使用、被 Agent 看见、被命令路由或被定时任务触发的对象，都先进入能力目录，再由 Bot 挂载引用。Bot 配置不直接嵌入不同能力类型的完整定义。

### 3.1 能力目录与引用

应用内形成统一能力目录：

```text
Capability
├── Skill
├── MCP Server
├── Custom App
├── Suite
├── Command
├── Scheduled Task
└── Future Capability Types
```

能力定义和 Bot 挂载引用分离：

```ts
type CapabilityKind = "skill" | "mcp" | "app" | "suite" | "workflow" | "command" | "scheduled-task";

interface CapabilityDefinition {
  kind: CapabilityKind;
  id: string;
  name: string;
  description: string;
  source: "builtin" | "local" | "market" | "config";
  enabled: boolean;
  version?: string;
  tags?: string[];
}

interface BotCapabilityRef {
  kind: CapabilityKind;
  id: string;
  enabled: boolean;
  policy?: CapabilityPolicy;
  alias?: string;
}

interface CapabilityPolicy {
  allowAgentUse?: boolean;
  allowCommandUse?: boolean;
  allowScheduledUse?: boolean;
  requireOwnerApproval?: boolean;
}

interface BotConfig {
  // 兼容旧配置；迁移后仍可由 resolver 转换为 skill refs。
  skillNames: string[];
  capabilityRefs: BotCapabilityRef[];
}
```

兼容策略：`skillNames` 保留并继续作为旧版 Skill 授权来源，`CapabilityResolver` 在运行时把它转换成 `kind=skill` 的引用。新增 `capabilityRefs` 缺省为空。旧配置迁移时不自动授权任何 MCP、自定义应用、套件、命令或定时任务。

### 3.2 三层结构

能力治理分三层实现：

1. Capability Registry：发现和加载能力定义，包括 Skills、MCP、自定义应用、套件、命令和后续扩展类型。
2. Capability Resolver：按 Bot 的挂载引用、启用状态、来源优先级和 policy 解析出本次可见能力。
3. Runtime Binding：把解析后的能力转换为具体运行时输入，例如 Skill symlink、Claude MCP 配置、Agent prompt、命令目标或定时任务执行目标。

这三层必须分开。新增能力类型时，优先新增 registry handler 和 runtime binding，不修改消息处理主流程，也不在 `BotConfig` 上继续增加平铺字段。

其中 Runtime Binding 再细分为两步：

1. `ExecutableCapabilityBinding Resolver`：把 `kind=id` 的能力引用解析为可执行绑定，例如单 Skill Claude 运行、套件上下文 Claude 运行或自定义应用运行。
2. `Capability Executor`：消费已解析好的 binding，真正执行 Claude Agent 或自定义应用入口。

这样命令、定时任务和后续 workflow 都只依赖 binding，不直接理解底层是 Skill、套件还是自定义应用。

### 3.3 Bot 挂载边界

Bot 挂载的是能力引用，不是能力所有权。一个 Skill、MCP 或套件可以被多个 Bot 挂载，但运行时必须按 Bot 重新解析：

- Bot A 挂载某套件，不代表 Bot B 可见该套件。
- 套件引用某 Skill/MCP/自定义应用，不代表 Bot 自动获得这些子能力；子能力必须也在 Bot 授权范围内，或由 UI 在挂载套件时显式展开并确认。
- 命令和定时任务只能调用当前 Bot 已挂载且 policy 允许的能力。
- Agent prompt 只能看到 resolver 输出的能力摘要，不读取全局能力目录。

### 3.4 扩展治理诊断

能力页提供统一扩展治理诊断，避免自定义应用、套件和 Workflow 只“可导入”但不可判断风险：

- 自定义应用：检查 `entry.type`、网络权限、文件系统权限、Owner 审批声明和可调用能力声明。`executable`、网络访问或超出 workspace/session/bot-state 的文件权限会标记为高风险。
- 套件：检查引用的 Skill、自定义应用和 MCP 是否存在；套件不会因为被挂载而自动授予这些子能力。
- Workflow：检查步骤中的 capability 引用是否存在；缺失依赖会阻止其作为命令或定时任务的可靠目标。
- 诊断状态分为 `OK/WARN/ERROR`，并给出授权前建议。导入资源不等于授权给 Bot，管理员应先处理 ERROR/WARN 项再授权。

Owner 审批不是只在 UI 中提示。运行时执行命令或定时任务前会检查 `capabilityRefs[].policy.requireOwnerApproval` 和自定义应用 `permissions.requiresOwnerApproval`。命令命中审批策略时会创建 Owner 私聊审批请求；定时任务命中时失败并写入运行历史，避免无人值守任务绕过审批。

### 3.5 目录建议

```text
workspace/
├── skills/
├── market-skills/
├── apps/
├── suites/
└── bots/<bot-id>/sessions/<conversation-hash>/

state/
└── bots/<bot-id>/
    ├── scheduled-runs.jsonl
    └── capability-cache.json
```

`capability-cache.json` 只缓存 resolver 结果和诊断信息，可重建，不作为授权来源。授权来源始终是应用配置。`capability-audit.jsonl` 保存命令、定时任务和审批阻断的能力使用审计，按 Bot 隔离，只用于治理报表。

### 3.6 自定义应用

自定义应用是一种可导入、可授权、可由命令或定时任务触发的代码能力。它用于承载未来的“小应用”“行业工具”“内部自动化脚本”或“带 UI 的本地工具”，但必须遵循统一 manifest 和运行协议。

导入后的目录建议位于：

```text
workspace/apps/<app-id>/
├── app.json
├── README.md
├── bin/
├── dist/
└── assets/
```

`app.json` 示例：

```json
{
  "id": "daily-quality-report",
  "name": "每日质量报表",
  "description": "生成并发送质量日报",
  "version": "1.0.0",
  "entry": {
    "type": "node",
    "command": "node",
    "args": ["dist/index.js"]
  },
  "interface": {
    "input": "json",
    "output": "json"
  },
  "capabilities": {
    "agentCallable": true,
    "commandCallable": true,
    "scheduledCallable": true,
    "hasUi": false
  },
  "permissions": {
    "network": false,
    "filesystem": ["workspace"],
    "requiresOwnerApproval": false
  }
}
```

首版建议支持两类 entry：

- `node`：使用应用内置 Node/Electron 运行 JS 入口，适合自包含脚本。
- `executable`：运行随应用包导入的可执行文件，但必须在导入时显式提示风险。

后续可以扩展：

- `webview`：本地 HTML/React 小应用，由 UI 打开，不直接进入 Agent。
- `mcp-adapter`：把自定义应用包装为 MCP server。
- `workflow`：只包含声明式步骤，由 Runtime 调度其他能力执行。

运行协议首版应保持简单：

```json
{
  "botId": "bot-a",
  "trigger": "command|scheduled|agent",
  "input": "用户参数或任务 prompt",
  "context": {
    "messageId": "可选",
    "chatId": "可选",
    "workspace": "当前 Bot 会话 workspace"
  }
}
```

应用输出：

```json
{
  "ok": true,
  "reply": "可直接回复给用户的文本",
  "artifacts": []
}
```

自定义应用的安全边界：

- 自定义应用必须作为 `kind=app` 进入 Capability Registry，再由 Bot 通过 `capabilityRefs` 授权。
- 命令和定时任务只能调用当前 Bot 已挂载且 policy 允许的自定义应用。
- 自定义应用默认不能访问其他 Bot 状态、其他 Bot workspace、全局 Skills 或全局缓存。
- 导入应用不等于授权给任何 Bot；授权时需要展示 manifest 中的入口、权限和风险。
- 应用不能把 API Key、Token 或用户数据写入仓库；敏感配置只能进入本机 config 或 Bot 状态目录。

### 3.7 套件

套件是面向角色或行业的能力包，不直接执行代码。套件可以包含：

- `skills`: Skill 名称列表。
- `mcpServers`: MCP 服务 ID 列表。
- `apps`: 自定义应用 ID 列表。
- `workflows`: 工作流说明或提示模板。
- `instructions`: 给 Agent 的角色、行业、工具选择和输出约束。
- `commands`: 推荐命令模板。

套件来源首版建议支持两类：

- 内置套件：随应用打包，位于 `builtin-suites/`。
- 用户套件：用户通过 GUI 导入目录，复制到 `workspace/suites/`。

Git 市场套件可复用 Skill 市场机制，但建议放在 2.0.0 后续迭代，避免和 MCP 配置、命令机制同时扩大供应链面。

套件目录建议结构：

```text
suite.json
README.md
workflows/
```

`suite.json` 示例：

```json
{
  "id": "manufacturing-qa",
  "name": "制造业质量问答",
  "description": "面向质量追溯、异常分析和文档检索的工具套件",
  "version": "1.0.0",
  "publisher": "BlackLake",
  "trusted": true,
  "tags": ["quality", "manufacturing"],
  "skills": ["moje-qa-assistant-basic", "docx"],
  "mcpServers": ["quality-db"],
  "apps": ["daily-quality-report"],
  "instructions": "优先检索质量知识库，再结合飞书资料回答。",
  "workflows": [
    {
      "id": "root-cause-analysis",
      "name": "异常根因分析",
      "prompt": "按现象、批次、工序、可能原因和建议动作组织回答。",
      "steps": [
        {
          "id": "collect-facts",
          "name": "收集事实",
          "type": "prompt",
          "prompt": "先整理用户输入、附件和上下文里的事实，不做结论。"
        },
        {
          "id": "analyze",
          "name": "分析原因",
          "type": "capability",
          "prompt": "基于已收集事实分析可能原因。",
          "input": "{{stepPrompt}}\n\n事实：\n{{steps.collect-facts}}\n\n用户输入：\n{{input}}",
          "condition": { "if": "{{previous}}", "includes": "异常" },
          "repeat": { "maxTimes": 2, "until": { "if": "{{previous}}", "includes": "结论" } },
          "continueOnError": true,
          "timeoutSeconds": 120,
          "retry": { "maxAttempts": 3 },
          "capability": { "kind": "skill", "id": "moje-qa-assistant-basic" }
        }
      ]
    }
  ]
}
```

`version`、`publisher`、`trusted` 和 `tags` 用于能力页展示、来源可信度提示和后续套件市场治理。导入或升级套件时会在受管目录写入 `.qft-suite-state.json`，记录安装、升级和更新时间；卸载前会检查 Bot 授权、命令和定时任务引用。

`steps` 为可选字段。没有 `steps` 时，Workflow 按 `prompt` 作为强约束前缀执行；存在 `steps` 时，Runtime 会按顺序执行步骤，并把上一步输出传给下一步。首版步骤类型：

- `prompt`：在父套件上下文中执行一段 Claude Agent prompt。
- `capability`：调用已授权的 Skill、套件或自定义应用；当前不允许递归调用 Workflow。

步骤可选治理字段：

- `input`：步骤输入模板，支持 `{{input}}`、`{{previous}}`、`{{workflowPrompt}}`、`{{stepPrompt}}`、`{{steps.<stepId>}}` 和 `{{variables.<key>}}`，未设置时默认拼接 workflow prompt、step prompt、上一步输出和当前输入。
- `condition`：步骤执行条件，支持 `if` 模板与 `equals`、`includes`、`matches`、`not` 判断；不满足条件时写入 `skipped` 步骤事件。
- `repeat.maxTimes` / `repeat.until`：步骤循环执行上限和提前结束条件；`maxTimes` 会被限制在安全上限内。
- `continueOnError`：步骤重试耗尽后允许 Workflow 继续执行，用于可选增强步骤或降级分支。
- `timeoutSeconds`：单步超时时间；超时后该尝试失败。
- `retry.maxAttempts`：单步最大尝试次数；每次失败都会写入步骤事件，成功后继续下一步。

执行观测：

- 每个步骤发出 `started/success/failed/skipped` 事件。
- 运行台记录步骤状态、尝试次数、短输出摘要或错误摘要。
- 定时任务触发 Workflow 时，步骤摘要和尝试次数写入 `scheduled-runs.jsonl` 的 `detail` 字段。

## 4. 定时任务设计

### 4.1 数据模型

定时任务属于 Bot：

```ts
interface ScheduledTask {
  id: string;
  botId: string;
  enabled: boolean;
  name: string;
  schedule: {
    type: "interval" | "daily" | "weekly" | "cron";
    timezone: string;
    everyMinutes?: number;
    timeOfDay?: string;
    weekdays?: number[];
    cronExpression?: string;
  };
  target: {
    type: "agent" | "command" | "capability";
    commandName?: string;
    capability?: { kind: CapabilityKind; id: string };
    prompt: string;
  };
  delivery: {
    type: "chat";
    chatId: string;
    replyIdentity?: "bot" | "user";
  };
  retry?: {
    maxRetries: number;
    delayMinutes: number;
  };
}

interface ScheduledTaskRuntimeState {
  id: string;
  lastRunAt?: string;
  nextRunAt?: string;
  lastStatus?: "success" | "failed" | "skipped";
  failureCount?: number;
  retryAt?: string;
  pausedReason?: string;
}
```

状态文件：

```text
state/bots/<bot-id>/scheduled-tasks.json
state/bots/<bot-id>/scheduled-runs.jsonl
```

任务配置保存在 `config.json` 还是状态文件需要区分：

- 用户创建、编辑、启停的任务定义放入应用配置，便于备份和迁移。
- `lastRunAt`、`nextRunAt`、`lastStatus`、`failureCount`、`retryAt` 和 `pausedReason` 等运行态字段按任务 `id` 写入 Bot 状态目录，不回写任务定义。
- `scheduled-tasks.json` 新写入格式只保存运行态记录；读取时兼容旧版完整任务数组，只按 `id` 合并运行态字段，旧文件中的 schedule、target、delivery 等定义字段不会覆盖当前配置。
- 每次运行审计记录、错误和耗时继续写入 `scheduled-runs.jsonl`。

### 4.2 调度器

Runtime 初始化时为每个 Bot 加载任务并计算 `nextRunAt`。调度器只负责在本机应用运行时触发：

- `interval`、`daily`、`weekly` 覆盖常见配置，`cron` 使用 5 段表达式覆盖更复杂的日历计划。Cron 当前支持 `*`、列表、范围和步进，不支持秒级、`L/W/#` 或英文月份/星期。
- Bot 停用时不触发任务。
- Bot 监听未启动时，任务仍可执行，但必须满足模型配置完整；是否需要飞书回复取决于 `delivery.chatId`。
- 同一 Bot 的同一任务不可重入；上次仍在运行时本次记为 `skipped`。
- 定时任务进入现有 `TaskLimiter`，和飞书消息共享并发上限。
- 定时任务使用独立 conversation key，例如 `scheduled:<task-id>`，以便保留任务上下文但不污染用户私聊/群聊会话。
- 已保存且启用的任务可由用户在 Bot 编辑器中手动立即运行；手动运行复用相同执行链路和 `scheduled-runs.jsonl` 审计记录，并保留原本已计算的 `nextRunAt`。
- 计划触发失败时，如果任务配置了 `retry.maxRetries > 0`，调度器按 `retry.delayMinutes` 设置 `retryAt`，并优先于原计划触发。连续失败超过上限后写入 `pausedReason` 并停止自动排期；手动运行不消耗重试次数，成功运行会清空失败计数、`retryAt` 和暂停原因。

### 4.3 Agent 可调用面

Agent 只能管理当前 Bot 的任务。提供结构化结果协议，Runtime 解析后执行：

```text
SCHEDULE_TASK: {"name":"...", "schedule":{...}, "target":{...}, "delivery":{...}}
UPDATE_TASK: {"id":"...", "enabled":false}
LIST_TASKS: {}
```

首版更稳妥的实现是：Agent 输出结构化请求，Runtime 只在必要字段完整且目标在当前 Bot 授权范围内时落库。涉及 chatId 的任务创建必须来自当前会话，默认投递回当前 chat；跨 chat 投递需要 Owner 授权或 UI 配置。

## 4.4 下载前缓存命中

2.0.0 需要把现有 `state/file-cache/<sha256>/` 从“下载后归档”升级为“下载前可命中的内容缓存”。目标场景：同一个 Bot 在同一个群里，不同用户围绕同一个大型 PPT 或飞书文件提问时，第二次处理应优先复用已缓存文件，而不是重新从飞书下载 500 MB 文件。

当前已实现飞书消息附件路径：`downloadMessageResources` 会在下载前按当前 Bot 和消息资源 key 查询缓存，命中后将缓存文件复制到当前消息目录。Agent 下载云盘文件或导出云文档的路径已接入首版受控 helper：Agent 输出 `LARK_CACHED_FILE` 后由 Runtime 在主进程中查缓存、下载或导出、写入缓存，并把当前会话本地路径回灌给 Agent 继续分析。Runtime 会拦截 Agent 通过 Bash 裸调 `lark-cli drive +download` 或 `drive +export` 的路径，引导改走受控文件缓存 helper。

### 4.4.1 当前限制

旧实现会在消息附件或 Agent workspace 文件下载完成后写入应用级内容哈希缓存，并在 metadata 中记录关联 `botIds`，但下载前没有按飞书文件标识查缓存。当前消息附件、受控云盘下载和受控云文档导出已补齐下载前缓存命中；缺少远端新鲜度字段的导出缓存会标记为 unknown 并按治理策略重新获取或提示风险。

### 4.4.2 缓存索引

文件索引：

```text
state/file-cache/index.json
```

索引键候选：

- 消息附件：`lark-message-resource:<bot-id>:<file-key>`。
- 飞书云盘文件：`lark-drive-file:<bot-id>:<file-token>`。
- 飞书云文档导出：`lark-drive-export:<bot-id>:<doc-type>:<file-token>:<revision-or-updated-time>:<extension>`。

索引值指向内容哈希目录：

```json
{
  "cacheKey": "lark-drive-export:bot-a:slides:xxx:2026-06-16T10:00:00Z:pptx",
  "hash": "sha256...",
  "fileName": "example.pptx",
  "bytes": 524288000,
  "botIds": ["bot-a"],
  "source": {
    "type": "lark-drive-export",
    "fileToken": "xxx",
    "docType": "slides",
    "updatedAt": "2026-06-16T10:00:00Z"
  }
}
```

### 4.4.3 复用规则

- 只允许当前 Bot 复用 metadata 中包含自身 `botId` 的缓存。
- 私聊和群聊的会话隔离不阻止同 Bot 文件缓存复用；缓存复用边界是 Bot 授权，不是 sender。
- 不同 Bot 即使文件 hash 相同，也不能在未记录授权前直接复用。
- 云文档导出必须包含 revision、updated time 或等价新鲜度字段；无法确认新鲜度时重新下载。
- 缓存命中后仍要把文件复制或链接到当前会话 workspace，避免 Agent 直接访问全局缓存目录。
- 清理会话不删除文件缓存；清理全部文件缓存会删除 index 和内容目录。存储管理也支持按索引 `cacheKey` 删除单条缓存；若同一内容 hash 仍有其他索引引用，内容目录会保留。
- 新写入的缓存索引记录 `cachedAt`。存储统计会自动清理超过 90 天且有明确缓存时间的索引；如果同一内容 hash 仍被其他索引引用，内容目录继续保留。旧索引缺少 `cachedAt` 时会尝试从 metadata 回填，仍无法判断时间的旧缓存不会被自动删除。

### 4.4.4 实施位置

- `lark-cli.ts` 的消息资源下载前先查附件缓存索引；未命中时下载并写入索引。
- Agent 通过 `LARK_CACHED_FILE` 受控协议下载云盘文件或导出云文档时，会先走主进程缓存命中；Agent 裸调原始 `lark-cli drive +download/+export` 会被 Runtime 拦截并引导改用受控协议。
- `file-cache.ts` 负责写入内容哈希、metadata 和索引，并提供 `lookup`、`materializeToWorkspace`、单条索引删除和全量 `clear`。

## 5. MCP 配置设计

### 5.1 配置模型

```ts
interface McpServerConfig {
  id: string;
  name: string;
  enabled: boolean;
  transport: "stdio" | "http" | "sse";
  command: string;
  args: string[];
  url?: string;
  env: Array<{ name: string; value: string; secret?: boolean }>;
  cwd?: string;
  description?: string;
}
```

当前可保存 `stdio`、HTTP 和 SSE。`stdio` 已接入运行时、命令/定时目标和协议探测；HTTP/SSE 只作为占位配置保存 URL，并在能力页提示运行时注入和协议探测尚未接入。远端 MCP 涉及网络身份、证书、可用性和权限审计，正式运行前仍需单独设计。

MCP 服务本身是全局能力定义，Bot 通过 `capabilityRefs` 挂载 `kind=mcp` 的引用。敏感 env 仍只存本机配置，不进入 Git。

### 5.2 Agent 接入

MCP 只在 Bot 授权后传给 Claude Agent SDK。当前 `runClaude` 只会把已启用、已授权且传输类型为 `stdio` 的 MCP 构建为 SDK 配置，并在 prompt 中明确：

- 只能使用当前 Bot 授权的 MCP。
- MCP 工具和 Skill 一样受当前 Bot 权限约束。
- 不得尝试读写其他 Bot 的状态和 workspace。

HTTP/SSE MCP 即使已保存 URL，也不会进入 Agent、命令和定时任务目标，直到远端运行时接入完成。

### 5.3 安全边界

- MCP command 不允许来自聊天消息即时创建并执行。
- UI 新增 MCP 前展示风险提示：MCP 服务等价于本机工具能力。
- MCP 进程环境变量只注入该服务声明需要的字段。
- MCP 进程 cwd 默认位于当前 Bot workspace 或用户显式选择的安全目录。
- 套件挂载 MCP 时，只表达推荐关系；Bot 未授权的 MCP 不会因套件存在而自动可用。

### 5.4 诊断与工具预览

MCP 诊断分为静态检查和协议探测：

- 静态检查覆盖启用状态、传输类型、stdio 命令解析、HTTP/SSE URL 缺失、cwd 可读性、环境变量缺值和 Bot 授权关系，页面初始加载即可执行。
- 协议探测只在用户点击刷新诊断时执行。应用短暂启动静态检查通过的 `stdio` MCP，发送 `initialize` 和 `tools/list`，展示最多 20 个工具名；失败时展示协议错误、退出码、signal 和 stderr 尾部，然后关闭进程。
- HTTP/SSE 不做协议探测，只展示“已可保存配置，但运行时注入和协议探测尚未接入”的占位诊断。
- 每次协议探测会把服务 ID、服务名、探测时间、状态、耗时、工具名摘要、错误、退出码、signal 和 stderr 尾部追加到 `state/mcp-diagnostics.jsonl`。页面静态诊断会读取最近一次探测结果，便于重启应用后继续排查。
- 探测结果只用于配置可观测性，不自动授权 MCP，也不长期持有 MCP 进程。

## 6. 命令机制设计

### 6.1 命令匹配

命令格式：

```text
/xxxx [参数文本]
```

规则：

- 只匹配消息 trim 后以 `/` 开头的第一个 token。
- `/new`、`/continue`、`/owner` 是系统保留命令，优先级高于用户命令。
- 命令名称只允许小写字母、数字、短横线和下划线，避免和普通文本误判。
- 未配置命令时按普通 Agent 消息处理，除非命中系统保留命令。

### 6.2 命令绑定

首版命令绑定支持 Skill、套件和自定义应用。绑定目标统一使用 capability 引用，避免命令层了解具体能力类型：

```ts
interface BotCommandBinding {
  name: string;
  aliases?: string[];
  enabled: boolean;
  description: string;
  target: {
    type: "capability";
    capability: { kind: "skill" | "mcp" | "suite" | "workflow" | "app"; id: string };
  };
  promptTemplate?: string;
}
```

执行语义：

- `/help` 是系统保留命令，会自动列出当前 Bot 已启用命令、别名和说明。
- Skill 命令：本次 Agent prompt 明确“优先且只能按该 Skill 的任务边界处理”，但仍保持 sandbox 和 Bot 授权校验。
- MCP 命令：本次 Agent prompt 聚焦到目标 MCP 服务，并仍只注入当前 Bot 已授权的 stdio MCP 配置；HTTP/SSE MCP 在运行时接入前不出现在命令目标中。
- 套件命令：本次 Agent prompt 注入套件 instructions、workflows 和套件内已被 Bot 授权的 Skill/MCP。
- Workflow 命令：复用父套件上下文，若 Workflow 声明 steps 则按步骤顺序执行。
- 自定义应用命令：Runtime 按 app manifest 启动应用入口，把命令参数和当前 Bot 上下文作为 JSON 输入；如果应用返回 `reply`，直接回复用户，也可按配置把结果交回 Agent 二次总结。
- 命令参数作为用户意图追加到 prompt，不做 shell 拼接。

### 6.3 自主决策与命令的关系

默认聊天仍交给模型在当前 Bot 授权的 Skills、MCP、自定义应用和套件中自主选择。命令是显式路由：

- 用户知道要走哪个入口时，用命令减少歧义。
- Agent 自主选择时，套件作为上下文和能力分组提示，不强制只能选一个。
- 多个套件同时挂载时，prompt 要列出套件名称、说明、包含能力和适用场景，让模型选择。

## 7. 主题切换设计

配置模型：

```ts
interface UiConfig {
  theme: "system" | "light" | "dark";
}
```

实现方式：

- `config/default.json` 增加 `ui.theme = "system"`。
- 渲染层启动时读取配置并设置 `document.documentElement.dataset.theme`。
- `system` 模式使用 `prefers-color-scheme`，并监听系统变化。
- CSS 使用变量表达颜色、边框、背景和阴影，避免为黑白两套复制布局规则。

UI 入口放在配置页全局设置区，不放到 Bot 编辑弹窗，因为主题是应用级偏好。

## 9. 建议实施顺序

1. 能力抽象底座：新增 `electron/capabilities.ts`，定义 `CapabilityDefinition`、`BotCapabilityRef`、registry、resolver 和 runtime binding 接口。
2. 配置模型与迁移：扩展 `AppConfig`、`BotConfig`、默认配置和迁移测试，旧 `skillNames` 通过 resolver 兼容，不再新增 `bot.xxxIds` 平铺字段。
3. 自定义应用发现：新增 `electron/apps.ts` 和 `workspace/apps/`，先支持 manifest 校验、导入、预览和 Bot 授权。
4. 命令解析器：先实现保留命令优先、用户命令匹配和 capability 目标校验，命令目标通过 resolver 查找。
5. 套件发现与授权：新增 `electron/suites.ts`、`builtin-suites/` 和 `workspace/suites/`，套件通过 registry 进入能力目录。
6. Agent prompt 扩展：把当前 Bot resolver 输出的 Skill、MCP、自定义应用、套件和命令注入 `runClaude`。
7. 定时任务存储与调度：复用 `TaskLimiter` 和会话队列，任务目标只允许 resolver 输出中 policy 允许定时调用的能力。任务定义字段和 `failureCount/retryAt/pausedReason/lastRunAt/nextRunAt/lastStatus` 等运行态字段已拆出明确 merge helper，避免旧状态文件或回滚升级时互相污染。
8. MCP 配置 UI 与运行时接入：先 stdio，MCP 通过 registry 定义、通过 `capabilityRefs` 授权，再接入 SDK。
9. 主题切换：独立改 UI 和 CSS，风险较低，可穿插实施。
10. 文档与版本：2.2.0 已完成版本记录、arm64 打包和归档验证；形成具体版本号后必须同轮执行 arm64 打包和验证。后续新变更继续进入 `Unreleased`。

## 10. 需要新增或调整的测试

- 配置迁移：旧配置缺省套件、MCP、命令、定时任务和主题字段。
- 命令解析：系统命令优先、未知命令回退、capability 授权校验。
- 自定义应用：manifest 校验、导入冲突、入口缺失、Bot 未授权不可调用、输出协议解析。
- 套件发现：内置/用户套件、缺失 Skill/MCP/自定义应用的提示、重复 ID 处理。
- 定时任务：nextRun 计算、禁用 Bot 不触发、运行中跳过、失败记录。
- Sandbox/MCP/应用：未授权 MCP 和自定义应用不进入 Agent 配置，运行时不扩大其他 Bot 目录访问。
- UI 构建：配置页新增入口、主题 class/data attribute 切换不破坏现有渲染。

## 11. 发布边界

2.0.0 是主版本，因为它改变了应用能力模型：Bot 不再只是授权 Skills，而是可挂载自定义应用、套件、命令、MCP 和定时任务。写入正式版本记录时，必须同轮完成：

1. `npm test`
2. `npm run pack:mac`
3. 核对 arm64 DMG/ZIP 文件名、应用版本和主程序架构
