## 2. 产品概述

QuarkfanTools 是一个运行在 macOS 本机的多机器人 Skill Agent 工具。用户在本机配置模型、飞书机器人和能力授权后，即可让飞书机器人在私聊或群聊中调用本地 Skills、套件、Workflow、自定义应用、MCP 服务和飞书知识能力完成问答、文档处理、自动化任务和定时推送。

产品的核心价值是：把“会使用飞书的业务用户”和“本机可控的 AI Agent 能力”连接起来，同时保持多机器人隔离、能力授权可治理、数据留在本机、安装包自包含。

当前正式开放的消息平台是飞书。企业微信 Provider 的配置兼容层、代码和官方 CLI 参考保留，但因官方能力限制，当前版本在 UI 和运行时暂时封闭；钉钉仅作为后续占位能力展示。

## 4. 产品演进历史

本节帮助新接手产品经理理解产品为什么演进成当前形态。逐版本明细以 [`../CHANGELOG.md`](../CHANGELOG.md) 为准，这里按产品阶段整理。

### 4.1 1.0：多飞书机器人与 Skill 授权雏形

`v1.0.0` 建立了最早的基本盘：支持配置多个飞书机器人，按机器人授权 Skills，机器人收到消息后添加处理中表情，任务结束后移除。这个阶段验证的是“飞书 Bot 作为入口，本机 Agent 调用授权能力处理消息”的核心闭环。

### 4.2 1.1-1.3：连续会话、文件处理、Skill 市场和运行稳定性

| 版本 | 产品变化 |
| --- | --- |
| `v1.1.0` | 增加 24 小时连续会话、图片多模态处理，Agent 默认可调用当前机器人身份下的飞书 CLI |
| `v1.2.0` | 内置 Word、PowerPoint、Excel Skills；PPT 结合多模态和 macOS Quick Look；会话 workspace 独立化；加入存储统计和清理 |
| `v1.3.0` | 新增 HTTPS Git Skill 市场、选择性清理、Office 内置 ZIP/XML 预处理，不再依赖 Office/Python/Node/LibreOffice |
| `v1.3.1` | 建立需求、架构、运维、安全、决策、状态文档；修复残留或重复飞书订阅导致的重连失败；应用限制为单实例 |
| `v1.3.2` | 扩大 macOS 窗口拖拽区域，改善桌面应用基础体验 |

这一阶段形成了三个重要产品原则：安装包自包含、会话可管理、文档可接续。

### 4.3 1.4-1.6：管理台、权限、Owner、缓存和现场可用性

`v1.4.0` 到 `v1.6.6` 是 1.x 系列的成熟阶段，重点从“能跑”转向“能被管理员配置、排障和交付”。

| 主题 | 历史变化 |
| --- | --- |
| 多 Bot 管理 | 运行台改为每个注册机器人独立启停；点击机器人可查看独立日志并按等级筛选 |
| Skill 授权治理 | 导入 Skill 只进入本地市场，默认不授权；授权支持搜索、批量操作、来源筛选和预览 |
| 版本与文档 | 侧栏增加可点击版本号和用户更新记录；维护版本规则和面向用户的更新说明 |
| 性能与排队 | 添加处理中表情与 Agent 并行；记录资源处理、Agent、飞书回复和总耗时；修复跨会话并发上限 |
| Owner 机制 | Bot 可配置 Owner，Agent 无法解决或需要人工授权时私聊 Owner 并回传结论 |
| 飞书资料访问 | 用户态 OAuth 统一从配置页发起；新增文档搜索权限、额外 scope、OAuth 摘要日志 |
| Sandbox 与 CLI | 修复 lark-cli trustd、锁文件、OAuth 安全存储等 sandbox 问题 |
| 文件缓存 | 新增飞书延后下载任务、内容哈希缓存、下载前命中和会话详情 |
| UI 手册 | Logo 入口打开应用内使用手册；左下角状态改为在线、监听、排队聚合 |

1.x 系列最终定位为“飞书多 Bot Skill Agent”的稳定基础线。当前 1.x 分支已封版，只作为历史兼容样本，不再作为同步目标。

### 4.4 2.0：从 Skill Agent 升级为能力治理平台

`v2.0.0` 是产品结构变化最大的版本。它把原先“Bot 授权 Skills 后由 Agent 自主选择”的模式，升级为“Bot 挂载统一能力，Agent、命令和定时任务在同一授权边界内调用”。

核心变化包括：

- 新增 Bot capability refs 和 Capability Registry/Resolver。
- 新增 `/xxx` 命令机制，命令可配置到 Skill、套件、Workflow、自定义应用等目标。
- 新增 Bot 级定时任务，支持本机调度、运行记录、投递 chat 和手动立即运行。
- 新增套件能力，可导入 `suite.json`，在 Bot 中挂载，并向 Agent 注入套件说明和已授权子能力摘要。
- 新增 Workflow，支持声明式步骤编排。
- 新增 MCP 能力，支持 `stdio` MCP 配置、Bot 授权、Claude Agent SDK 注入和诊断。
- 新增自定义应用能力，可导入 `app.json`，进入能力页预览和授权。
- 新增受控飞书文件缓存协议 `LARK_CACHED_FILE`，拦截裸调 `lark-cli drive +download/+export`。
- 新增多 IM Provider 底座，预留企业微信、钉钉、飞书知识连接器和结果投递路由。
- 新增主题切换、会话结构化时间线、长任务提示、资源目录打开等体验能力。

2.0 也同步修复了多飞书 Bot 群聊路由问题：群聊消息优先按 mention 目标匹配 Bot，不能用事件头 App ID 或 mention open_id 做错误排他。

### 4.5 2.0.1-2.0.3：飞书多 Bot 现场修复与凭据隔离

`v2.0.1` 到 `v2.0.3` 聚焦飞书现场问题和 Bot 凭据隔离：

- 修正群聊 mention open_id 与 bot info open_id 不一致时的路由策略。
- 用户态 OAuth 完成日志改为摘要，并明确 OAuth 不等于飞书 Bot 对群成员开放。
- 新增飞书 Bot 可用范围排障说明。
- 飞书用户态 OAuth、lark-cli 降级密钥和加密凭据改为 Bot 专属 HOME。
- 修复旧凭据 marker 导致 `invalid_client / The auth method is not supported` 的升级问题。
- 启动监听时增加“收到启动请求”“正在确认飞书 Bot 身份”等可见日志，并给短命令 30 秒超时。

这部分历史说明：飞书权限问题分为“本地 OAuth 能否读资料”和“飞书开放平台是否允许群成员使用 Bot”两类，产品需要在 UI 和排障手册里持续区分。

### 4.6 2.1：2.0 能力的产品化收口

`v2.1.0` 到 `v2.1.2` 主要把 2.0 的底层能力收成用户能理解和操作的产品：

- 发布链路收敛为 arm64 / Apple Silicon。
- 旧 `qah` 应用数据迁移前新增本地备份。
- 飞书 Provider 改为每个 Bot 独立 HOME/profile 维护事件订阅。
- 能力页新增扩展治理诊断、Bot 治理控制台和能力使用审计报表。
- 自定义应用和套件完成本机生命周期：导入、升级、卸载、引用保护、manifest 诊断。
- 命令新增别名、MCP 目标、`/help`、冲突提示和保留命令保护。
- 定时任务新增 5 段 cron、失败重试、失败告警、MCP capability 目标和独立定时任务中心。
- Workflow 新增输入模板、步骤输出变量、条件跳过、循环、失败恢复、超时和重试。
- MCP 新增 HTTP / SSE 占位配置、协议诊断日志和工具列表预览。
- 使用手册新增场景化教程，覆盖飞书问答、企业微信接入、命令、定时任务、套件/Workflow、MCP 和升级恢复。
- UI 对建设中能力给出友好响应，不再把未接入能力误列为可执行目标。
- 存储管理补齐文件缓存索引、90 天自动失效、新鲜度状态、索引修复、会话事件筛选和 JSON 导出。
- 自定义应用和套件预览升级为“说明 + Manifest 编辑器”，内置模板可复制为本地副本后继续编辑。

产品上，这一阶段解决的是“功能都有了，但用户不知道怎么用、怎么配、怎么判断风险”的问题。

### 4.7 2.1.3-2.1.7：企业微信探索与能力页信息架构调整

`v2.1.3` 到 `v2.1.7` 集中探索企业微信 Provider：

- 参考官方 `WecomTeam/wecom-cli`，本地缓存源码用于后续对照。
- 尝试内置轮询桥、聊天列表、轮询 Chat ID、缓存初始化和 MCP 配置缓存检测。
- 发现官方 `wecom-cli` 是调用型工具，不提供飞书式事件长连接；默认轮询要求指定 chat，产品体验和可维护性不足。
- 能力页改为多层级导航，把治理总览、MCP、套件、自定义应用、诊断与审计拆开，降低理解成本。

这段历史的结论是：企业微信能力不应在当前约束下继续半开放，否则会让用户误以为“已支持企业微信实时机器人”。因此后续在 `v2.2.0` 暂时封闭入口。

### 4.8 2.2：企业微信封闭与飞书 CLI 本机升级路径

`v2.2.0` 明确把企业微信 Provider 暂时封闭：

- 配置页禁用企业微信消息平台、CLI 缓存初始化、聊天列表获取、轮询配置和企业微信投递路由。
- 运行时拒绝启动历史企业微信 Bot。
- 能力页 IM / CONNECTORS 诊断提示先改用飞书。
- 企业微信历史配置、代码和官方 CLI 参考保留，方便后续有可维护方案时恢复。

`Unreleased` 中新增飞书 CLI 本机优先升级路径：

- Bot 显式 `cliPath` 优先。
- 其次检测本机可信 `lark-cli`。
- 最后回退安装包内嵌 runtime。
- Bot 隔离 HOME、profile、配置目录和 OAuth 状态不变。

这让现场可以升级飞书 CLI，而不需要等整个 QuarkfanTools 重新打包，同时仍保留新机器“拿来可用”的内嵌兜底。

### 4.9 历史决策摘要

| 决策 | 当前结论 | 原因 |
| --- | --- | --- |
| 1.x 是否继续维护 | 暂时不关注，分支封版 | 产品重点已转到 2.0 能力治理体系 |
| 当前发布架构 | 只默认交付 arm64 | 用户后续明确暂不考虑双架构；Intel 作为历史样本 |
| 企业微信 Provider | 当前封闭 | 官方 CLI 不提供稳定事件长连接，轮询指定 chat 体验不足 |
| 能力授权方式 | Bot 维度显式授权 | 防止导入新能力后自动扩大 Bot 权限 |
| 命令和定时任务 | 必须复用能力治理边界 | 避免通过命令或计划绕过 Skill/MCP/应用授权 |
| 自定义应用和套件 | 内置模板 + 本地编辑 | 用户不能靠猜 manifest，需要可学习示例 |
| 预留能力 | UI 友好提示建设中 | 避免看似可用但点击无反应或运行时报错 |
| 飞书 CLI 升级 | 本机优先，内嵌兜底 | 兼顾现场升级和自包含安装 |

## 5. 用户与角色

### 5.1 主要用户

| 用户 | 目标 | 典型行为 |
| --- | --- | --- |
| 业务使用者 | 在飞书里直接提问、处理文件、触发命令或接收定时结果 | @ 机器人、私聊机器人、上传 Office/图片、发送 `/help` 或业务命令 |
| Bot 管理员 | 配置机器人、模型、权限、能力、命令和定时任务 | 新增 Bot、填写 App ID/Secret、授权 Skills、配置命令和定时任务 |
| Skill / 能力维护者 | 编写或导入可复用能力 | 导入 Skill、自定义应用、套件，编辑 manifest，排查诊断 |
| 产品经理 | 管理产品范围、版本优先级、验收口径和用户体验 | 维护需求矩阵、定义新 Provider、评估商业化或行业模板 |
| 开发/交付人员 | 实现、打包、排障和客户现场支持 | 跑测试、打包 DMG、查看日志、处理授权和 CLI 问题 |

### 5.2 典型使用场景

1. 飞书群里 @ 业务助手，让它基于本地 Skill 和飞书知识回答问题。
2. 私聊机器人上传 Word、PPT、Excel 或图片，让 Agent 解析并生成结果。
3. 管理员给某个 Bot 授权指定 Skill、套件或 MCP，确保不同 Bot 的能力隔离。
4. 管理员为 Bot 配置 `/日报`、`/复盘` 等命令，让普通用户用稳定入口触发能力。
5. 管理员配置每日或每周定时任务，自动生成并投递日报、巡检或提醒。
6. 能力维护者从内置模板复制自定义应用或套件，逐步改造成业务能力。
7. 管理员在能力页查看 Bot 授权、命令绑定、定时任务绑定和能力使用审计。
8. 交付人员通过运行台日志、MCP 诊断、IM 诊断和存储管理排查问题。

## 6. 产品原则

- **拿来可用**：安装包必须自包含，不能要求客户预装 Git、Node、Python、Office、LibreOffice 或命令行工具。
- **本机优先**：配置、凭据、OAuth 状态、会话和用户导入能力都保存在本机。
- **机器人隔离**：不同 Bot 的 IM 凭据、OAuth、状态、会话、workspace 和能力授权必须隔离。
- **显式授权**：导入 Skill、自定义应用、套件或 MCP 不等于授权给 Bot；新增能力默认不可用。
- **能力可治理**：Agent、命令、定时任务和 Owner 审批共享同一 Bot 能力授权边界。
- **升级兼容**：旧数据尽量兼容迁移；确实不能兼容时，迁移前必须留下本机备份。
- **预留能力要友好**：尚未完成的能力必须在 UI 中明确展示“建设中”或“暂时封闭”，不能静默失败。

### 8.3 飞书 CLI 运行时与升级

安装包内嵌 arm64 `lark-cli`，普通用户无需安装命令行工具。高级现场需要升级飞书 CLI 时，运行时按以下顺序选择：

1. Bot 显式配置的 `cliPath`
2. 本机已安装的可信 `lark-cli`
3. 安装包内嵌 `runtime/lark-cli`

## 10. 安全与权限边界

### 10.1 数据与凭据

- 模型 API Key、飞书 App Secret、OAuth 状态、连接器凭据和用户数据仅保存在本机。
- 不得把 API Key、App Secret、Token、OAuth 状态或用户数据提交到 Git。
- 每个 Bot 使用独立状态目录、lark-cli HOME、OAuth 安全存储和 workspace。
- 会话清理不得删除应用配置、飞书授权或用户 Skills。

### 10.2 能力授权

- Skill、自定义应用、套件、MCP 导入后默认不授权给任何 Bot。
- Bot 只能看到明确授权给它的能力。
- 命令和定时任务不能绕过 Bot 授权和 capability policy。
- Owner 审批只代表人工处理结论，不会自动扩大 Bot 权限。
- 能力使用审计只读展示，不作为授权来源。

### 10.3 Agent 运行边界

- Agent 使用 Claude sandbox 和目录隔离限制文件访问。
- 当前 Bot 的 lark-cli 状态目录需要可读写，其他 Bot 状态和 workspace 不可访问。
- Runtime 会拦截 Agent 裸调 `lark-cli drive +download` 或 `drive +export`，引导走受控文件缓存 helper。
- 自定义应用和 MCP 都属于高风险扩展能力，必须显式授权和审查。

## 11. 升级与发布

### 11.1 升级兼容

- 旧版本配置尽量兼容读取和迁移。
- 从旧 `qah` 应用数据迁移到 `quarkfantools` 前，会备份到 `quarkfantools/backups/legacy-qah-<timestamp>/`。
- 新增字段必须有保守默认值，不得因升级自动扩大权限或启用新能力。
- 不兼容结构变更必须在本机留下备份。

### 11.2 发布约束

- 当前发布默认只考虑 arm64 / Apple Silicon。
- 每次形成版本号后，必须同步版本号、`CHANGELOG.md`、应用内更新记录、`STATUS.md` 和产物名称。
- 发布前至少执行 `npm test` 和 `npm run pack:mac`。
- 当前安装包未签名、未公证，不能把“成功打包”等同于“可无提示安装”。

## 13. 当前已知限制

- macOS 安装包尚未签名和公证。
- 当前正式开放飞书；企业微信暂时封闭，钉钉建设中。
- Skill 市场只支持 HTTPS Git，不支持 SSH。
- 会话过期固定为 24 小时，暂不能在 UI 中调整。
- 定时任务只在本机应用运行期间触发，不提供系统级后台常驻调度。
- MCP HTTP / SSE 只支持保存配置和占位诊断，不进入运行时。
- 自定义应用 `webview`、`mcp-adapter`、`workflow` 入口仍是建设中能力。
- 自动化测试覆盖大量配置和运行逻辑，但真实飞书端到端、真实 MCP 服务和 UI 自动化覆盖仍需专项补齐。
- PowerPoint 视觉预览依赖 macOS Quick Look，预览质量受系统支持影响。

## 14. 后续路线建议

1. 将 24 小时会话过期时间、文件缓存保留期和磁盘配额做成可配置项。
2. 增加更多行业套件和自定义应用模板，让用户从场景开始，而不是从 manifest 开始。

1. 自定义应用市场、签名校验和来源信任体系。
2. HTTP / SSE MCP 运行时接入。
3. 钉钉 Provider。
4. 工作流可视化编辑、步骤级重跑和专门运行历史页。
