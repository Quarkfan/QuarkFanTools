# 产品需求

## 1. 产品定位

QuarkfanTools 是面向非开发环境交付的 macOS 小工具。用户配置模型、飞书机器人和 Skills 后，即可让飞书私聊或群聊中的机器人通过 Claude Agent SDK 调度本地 Skills、知识和飞书能力处理消息。

## 2. 核心原则

- **拿来可用**：不能假定用户安装 Node、Python、Git、Office、LibreOffice 或其他基础环境。
- **双架构支持**：分别交付 Apple Silicon 和 Intel macOS 安装包。
- **机器人隔离**：不同机器人拥有独立凭据、状态、会话、workspace 和 Skill 访问范围。
- **用户可控**：用户决定机器人启停、Skill 授权、Skill 来源和会话数据清理。
- **可持续接续**：需求、架构、状态和版本变化必须保存在仓库文档中。

## 3. 功能需求

### 3.1 模型配置

- 支持配置兼容 Claude Messages API 和工具调用的第三方模型。
- 支持配置 Base URL、模型名、API Key 环境变量名和 API Key。
- 支持启用或禁用多模态能力。

### 3.2 多飞书机器人

- 用户可配置多个机器人，并独立启停。
- 运行台为每个已注册机器人提供独立启动和停止监听控制。
- 机器人仅在自身凭据和模型连接均配置完整时允许启动监听。
- 每个机器人可配置 App ID、App Secret、Bot 态或用户态收发身份。
- 同一时间运行的多个机器人必须对应不同飞书 App ID；同一个飞书应用下的多角色、多知识或多工作流应通过 Skill、命令或套件路由，不应配置为多个本地 Bot。
- 用户态支持在应用配置页发起 OAuth；Agent 不在聊天会话内要求普通用户扫码授权。
- 用户态 OAuth 必须申请飞书文档搜索权限；Agent 查找和读取飞书文档、Wiki、云盘与云 PPT 时使用用户态身份。
- 用户态 OAuth 不等同于飞书 Bot 对群成员的使用授权；群成员能否 @ 机器人由飞书开放平台应用发布状态和可用范围控制。
- 每个机器人可配置用户态 OAuth 额外 scope 列表；应用发起授权时必须与默认文档搜索权限合并，并要求用户重新授权后生效。
- 每个机器人必须拥有独立 lark-cli HOME、用户态 OAuth 安全存储、Claude home、会话 workspace 和 worker 进程；Agent sandbox 不得依赖或放行 macOS 用户全局 lark-cli 授权目录。
- 每个机器人在 UI 和权限上独立启停；`v1.7.0` 起，应用主进程只作为 Supervisor 管理多个 Bot worker，每个 worker 只承载一个 Bot 的飞书监听、消息处理、Agent、会话和去重。Docker 容器隔离作为可选增强能力预留，不得影响默认自包含交付。
- 群聊中包含飞书 `mentions` 元数据的消息必须先按被艾特机器人的 mention 目标过滤，再进入处理队列；mention 名称、App ID、应用名和 open_id 都可作为命中信号，但 mention open_id 不得作为否定其他命中的排他条件，避免多个机器人同时回复或正确机器人被误忽略。
- 多 Bot 严格群聊路由中，缺少 `mentions` 元数据的普通群消息默认不得仅凭事件 `sourceAppId` 触发 Bot 回复，避免刚 @ 后续未 @ 消息被不确定地处理。
- 可配置群聊免 @ 连续对话；默认关闭。开启后只能在同一群、同一发送者、刚 @ 的同一 Bot、指定时间窗口内免 @ 接续。
- 收到消息后在原消息添加处理中表情，处理结束后移除。
- 最终结果以飞书消息回复；Agent 可调用当前机器人身份下的 `lark-cli`。
- 每个机器人可配置一名有应用使用权限的 Owner；无法解决或需要人工授权时，向 Owner 私聊发送卡片，Owner 的处理结果回到原提问人。
- 每个机器人可选择向用户展示 Agent 的可观察工作进度；不得展示模型私有推理内容。

### 3.3 Skill 管理

- 自动发现安装包内置 Skills、Skill 市场 Skills 和用户导入 Skills。
- 用户可选择一个包含 `SKILL.md` 的文件夹，由应用复制到受管目录。
- 导入的 Skill 只进入本地 Skill 市场，默认不授权给任何机器人。
- 本地技能市场展示 Skill 名称、描述和来源，并允许删除用户导入的 Skill；删除时同步撤销对应机器人授权。
- 用户可配置 HTTPS Git 仓库、分支和可选 Token 作为 Skill 市场。
- 用户 Skill 在同名冲突时优先于市场和内置 Skill。
- 用户可按机器人勾选允许访问的 Skills；未授权 Skill 不可见、不可调用。
- Skill 较多时，机器人授权界面支持按名称或描述搜索，并对当前筛选结果批量授权或取消。
- 本地技能市场支持按来源和未授权状态筛选，并展示 Skill 已授权给哪些机器人。
- 点击技能市场中的 Skill 可预览 `SKILL.md` 和文件清单；导入同名 Skill 时必须明确报错，不得静默隐藏。
- 本地多个 Skill 的 frontmatter `name` 相同时，后续目录必须仍可在技能市场中以目录名区分显示。
- 正在被任一机器人授权使用的本地 Skill 不允许删除。
- 不支持全量通配授权；新增 Skill 必须由用户显式勾选后才能被机器人使用。
- Skill 可维护自身 `knowledge/`、`references/` 和 `scripts/`。
- Agent 执行前必须把当前机器人获授权的 Skill 复制物化到受管运行目录，不依赖 symlink 访问原始导入目录、市场仓库目录或安装包内置目录。
- Agent 会话 workspace 必须生成明确的 `CLAUDE.md`，列出当前 Bot 获授权 Skill 的 `./skills/<name>/SKILL.md` 入口，降低模型未主动读取 Skill 的不确定性。

### 3.4 消息、会话与多模态

- 支持飞书私聊和群聊艾特消息。
- 私聊按 chat 保持连续上下文；群聊按 chat 与发送者组合隔离上下文。
- 会话无活动 24 小时后过期。
- 用户可发送 `/new`、`新对话` 或 `重置会话` 主动创建新上下文。
- 图片消息自动下载并作为多模态输入。
- Word、PowerPoint、Excel 附件自动预处理；PowerPoint 在多模态开启时附带视觉预览。
- 找到高度匹配但下载耗时的飞书文件时，可先给出基本回答并创建待确认任务；用户通过 `/continue <任务编号>` 确认后继续下载和分析。

### 3.5 存储管理

- 展示可清理会话的占用、机器人、对话标识、更新时间和过期状态。
- 用户可点击会话查看 Claude 会话 ID、关联消息、最近对话记录和 workspace 文件清单。
- 支持清理过期会话、勾选会话或全部会话。
- 会话清理不得删除应用配置、飞书授权和用户 Skills。

### 3.6 自动化与定时任务

- 支持在 Bot 维度配置定时任务；任务必须继承当前 Bot 的身份、Skill 授权、Claude home、会话和并发限制。
- 支持间隔触发、一次性触发和 5 段 Cron 表达式触发，任务目标为 prompt，执行结果记录到运行台日志。
- 定时任务必须按 Bot 持久化到 `state/bots/<bot-id>/scheduled-tasks.json`，不得放入全局共享任务池。
- 任务应支持启停、手动运行、超时限制、错过策略和并发策略。
- 首版不主动发送飞书消息；未来接入飞书输出时必须显式配置输出目标。

### 3.7 UI 与可观测性

- GUI 支持配置、监听启停、运行状态、日志、Skill 列表、Skill 市场和存储清理。
- GUI 提供自动化页，按 Bot 管理定时任务。
- GUI 左上角 Logo 提供应用内使用手册，说明快速开始、配置项、Bot、Skill 市场、运行台、飞书权限和存储管理。
- GUI 全局状态应按多 Bot 聚合展示在线数量、监听数量和排队任务数，不使用单一 RUNNING/STOPPED 表示全部机器人。
- 机器人配置在配置页以列表展示，点击后通过弹窗编辑和保存详情；配置项应提供可点击说明入口。
- GUI 展示当前应用版本；点击版本号后显示面向用户的更新记录。
- 点击运行台中的机器人后展示其独立详细日志，并支持按日志等级筛选。
- 运行台支持一键复制诊断日志，内容包含当前运行快照、最近内存日志和持久化日志，方便用户反馈问题。
- 运行日志应记录飞书事件路由诊断、群聊免 @ 连续窗口命中情况，以及 Agent 本次可见的 Skill 上下文，便于区分未收到消息、路由忽略、Agent 未使用 Skill 和回复失败。
- 运行台应区分 Bot worker 已启动和飞书监听已连接，避免把“进程启动中”误判为“已经可以收消息”。
- 点击启动机器人监听后必须立即给出可见运行日志；启动前置阶段失败或超时不得静默无反馈。
- 运行台日志默认记录 Agent 可观察工作过程；是否向飞书用户发送进度消息由 Bot 配置单独控制。
- 日志记录消息接收、处理结果、错误、飞书事件投递延迟，以及资源处理、Agent、飞书回复和总耗时。
- 停止后再次启动监听必须可正常工作，不得残留重复监听状态。

## 4. 非功能需求

- 凭据和用户数据仅保存在本机，不进入 Git。
- macOS Agent sandbox 允许调用系统 trustd 完成内置 Go CLI 的代理 TLS 校验，但仍禁止 unsandboxed 命令并保持目录隔离。
- 单个对话内任务串行，跨对话并发受配置限制。
- 超出跨对话并发上限的消息进入处理队列，并在运行台展示排队数量。
- 单次 Agent 最大步数可配置，默认 60；复杂 Skill 或飞书资料检索不得被固定 20 步上限过早中断。
- 应用重启后可恢复未过期会话。
- Office 压缩文件解析必须限制条目数量和解压体积。
- 发布包包含运行所需的 Electron、Claude 运行时、飞书 CLI、Git 和 Office 解析能力。
- 可配置是否阻止系统自动休眠；该能力不能承诺阻止用户手动合盖、关机或强制睡眠。
- 系统从休眠唤醒后，应重建正在运行的 Bot 监听，降低长连接假连接风险。

## 5. 当前非目标

- 不支持 Windows 或 Linux 桌面端。
- 不支持 SSH Skill 市场地址。
- 不提供云端集中式机器人管理或用户数据同步。
- 不保证所有 PowerPoint 元素都能被 Quick Look 完整渲染。

## 6. 核心验收标准

1. 在未安装开发环境的新 macOS 上，配置模型和飞书机器人后可以启动监听并回复消息。
2. 停止监听后再次启动成功，不出现已有监听的残留错误。
3. 两个机器人只能访问各自授权的 Skills 和会话数据；群聊中艾特其中一个机器人时，其他机器人不得回复或占用处理队列。
4. 同一私聊在 24 小时内保持上下文；群聊中不同发送者互不共享上下文。
5. 收到消息后可见处理中表情，处理完成后表情被移除并收到最终答复。
6. Word、PPT、Excel 和图片消息可被处理，PPT 在多模态开启时使用视觉预览。
7. 选择性清理只删除目标会话，不影响配置、授权和 Skills。
8. 定时任务由对应 Bot worker 执行，不能越权访问其他 Bot 的 Skill 或会话。
