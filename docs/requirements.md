# 产品需求

## 1. 产品定位

QuarkfanTools 是面向非开发环境交付的 macOS 小工具。用户配置模型、IM 机器人和 Skills 后，即可让飞书机器人通过 Claude Agent SDK 调度本地 Skills、知识和外部资料能力处理消息。企业微信 Provider 的代码和配置兼容层保留，但因官方能力限制，当前 UI 和运行时入口暂时封闭。

## 2. 核心原则

- **拿来可用**：不能假定用户安装 Node、Python、Git、Office、LibreOffice 或其他基础环境。
- **Apple Silicon 优先**：后续发布和验证默认只交付 arm64 / Apple Silicon macOS 安装包；Intel x64 只作为历史版本兼容样本，不再作为当前发布目标。
- **兼容升级优先**：旧版本数据应尽量通过迁移和兼容读取适配；确实无法兼容的结构变化必须在迁移前留下本地备份，保证用户可回退或人工恢复。
- **机器人隔离**：不同机器人拥有独立 IM 凭据、连接器凭据、状态、会话、workspace 和 Skill 访问范围。
- **用户可控**：用户决定机器人启停、Skill 授权、Skill 来源和会话数据清理。
- **授权门禁**：应用启动和运行中必须检查远端授权开关；明确 `Auth=close` 时关闭，网络不可达按连续时长和次数累计后再判定。
- **可持续接续**：需求、架构、状态和版本变化必须保存在仓库文档中。

## 3. 功能需求

### 3.1 模型配置

- 支持配置一个或多个兼容 Claude Messages API 和工具调用的 MODEL PROVIDER。
- 每个 MODEL PROVIDER 支持配置 Provider ID、名称、Base URL、模型名、API Key 环境变量名、API Key、启停状态和是否支持多模态能力。
- 旧版单模型字段必须自动迁移为第一个 MODEL PROVIDER，并继续保留兼容字段，避免升级后旧配置失效。
- 系统设置必须独立于 MODEL PROVIDER，支持配置 MODEL PROVIDER 使用策略：轮流或随机。
- 系统设置必须支持失败切换开关；开启后，当前轮流或随机命中的 Provider 调用失败时，应按候选顺序尝试下一个完整 Provider；关闭时保持失败即报错。
- 视觉模型调用只能选择已启用、配置完整且开启多模态的 Provider；普通 Agent 和文本后处理调用只选择已启用且 Base URL、模型名、API Key 完整的 Provider。

### 3.2 多 IM 机器人

- 用户可配置多个机器人，并独立启停。
- 运行台为每个已注册机器人提供独立启动和停止监听控制。
- 机器人仅在自身凭据和模型连接均配置完整时允许启动监听。
- 每个机器人可选择消息平台。当前正式开放飞书；企业微信因官方能力限制暂时封闭入口，UI 必须禁用企业微信消息平台、缓存初始化、聊天列表获取、轮询配置和企业微信投递路由，并给出友好说明；钉钉作为后续 Provider 预留，UI 必须明确标记建设中并禁止启动监听。
- 企业微信 Bot 的历史配置、官方 CLI 适配代码和本地参考源码可保留，但当前版本不得启动企业微信监听、内置轮询、聊天列表获取、CLI 缓存初始化或企业微信结果投递；已有旧配置升级后必须显示“功能暂时封闭”，不能静默运行。
- 每个机器人可配置主消息平台凭据：飞书使用 App ID / App Secret；企业微信历史配置可保留智能机器人 Bot ID / Bot Secret，但当前不作为可启动凭据；飞书还可配置 Bot 态或用户态收发身份。
- 消息入口、知识连接器和结果投递路由必须分离；当前企业微信入口和企业微信投递路由暂时封闭，跨平台结构只保留为后续恢复时的兼容基础。
- 每个机器人可配置飞书知识连接器；当主消息平台不是飞书时，飞书文档、Wiki、云盘和云 PPT 能力只能通过该连接器启用。
- 每个机器人可配置结果投递路由，将最终回复复制发送到指定平台的 chat。
- 用户态支持在应用配置页发起 OAuth；Agent 不在聊天会话内要求普通用户扫码授权。
- 用户态 OAuth 必须申请飞书文档搜索权限；Agent 查找和读取飞书文档、Wiki、云盘与云 PPT 时使用用户态身份。
- 用户态 OAuth 不等同于飞书 Bot 对群成员的使用授权；群成员能否 @ 机器人由飞书开放平台应用发布状态和可用范围控制。
- 每个机器人可配置用户态 OAuth 额外 scope 列表；应用发起授权时必须与默认文档搜索权限合并，并要求用户重新授权后生效。
- 每个机器人在 UI 和权限上独立启停，异常断线自动重连。飞书 Provider 应为每个运行中的 Bot 使用独立 HOME/profile 维护自己的事件订阅，收到事件后再由 Runtime 按 Bot mention 目标路由到唯一目标 Bot；企业微信事件桥和后续钉钉 Provider 保持各自 Provider 边界。
- 每个飞书机器人必须拥有独立 lark-cli HOME 和用户态 OAuth 安全存储；Agent sandbox 不得依赖或放行 macOS 用户全局 lark-cli 授权目录。
- 飞书群聊中包含 `mentions` 元数据的消息必须先按被艾特机器人的 mention 目标过滤，再进入处理队列；mention 名称、App ID、应用名和 open_id 都可作为命中信号，但 mention open_id 不得作为否定其他命中的排他条件，避免多个飞书机器人同时回复或正确机器人被误忽略。
- 每个飞书机器人可单独开启“上下文免艾特回复 Beta”。默认关闭；关闭时群聊仍必须 @ 机器人。开启后，未 @ 的群聊消息只进入该 Bot 的 beta 职责判断流程，Agent 必须根据上下文、消息内容和自身授权能力判断是否需要回复；不明确属于自身职责时必须静默，不添加处理中表情或长任务提示。
- 飞书机器人应记录每个已看到 chat 的最后消息游标。断网或长连接中断恢复后，用户可在运行台对正在运行的飞书 Bot 手动触发“历史消息补处理 Beta”：应用从游标后拉取历史消息，按 Bot 配置的最大补处理条数排队进入现有处理队列，并继续复用消息去重、mention 路由和会话隔离规则。该能力依赖飞书历史消息接口、Bot 身份权限和已有游标，当前标记为 Beta。
- 收到消息后在原消息添加处理中表情，处理结束后移除。
- 最终结果默认回复到原消息平台；Agent 可在授权边界内调用当前机器人身份下的平台 CLI 或已配置的知识连接器 CLI。
- 每个机器人可配置一名有应用使用权限的 Owner；无法解决或需要人工授权时，向 Owner 私聊发送卡片，Owner 的处理结果回到原提问人。
- 每个机器人可选择向用户展示 Agent 的可观察工作进度；不得展示模型私有推理内容。
- 每个机器人支持配置定时任务；任务定义、启停、目标、失败重试、失败告警和投递 chat 必须在 Bot 维度独立管理。
- 定时任务最近运行记录必须在 GUI 中可查看，至少展示 Bot、任务、状态、开始/结束时间、耗时和失败或 Workflow 步骤摘要。
- 定时任务最近运行记录应支持按 Bot 和运行状态筛选。
- GUI 应提供定时任务中心，集中查看各 Bot 的任务、最近状态、上次执行时间、下次计划时间、手动运行入口和运行历史；每个任务应提供状态/日志入口，弹窗展示启用状态、计划、目标、投递、上次执行、下次执行、失败计数、重试/暂停原因和最近运行详情，避免用户盲等。立即执行入口不得只藏在列表深层，任务编辑弹窗和状态/日志弹窗也应提供执行已保存任务的按钮。
- 用户可在 Bot 定时任务列表中手动立即运行已保存且启用的任务，手动运行应进入同一运行历史审计。
- 定时任务应支持可配置失败重试；计划触发失败超过上限后应暂停自动排期并展示暂停原因。
- 定时任务只在应用运行期间调度，但应用启动或配置刷新时如果发现持久化 `nextRunAt` 已到期，应保留该到期时间并尽快追赶执行一次，而不是直接跳到下一次未来计划。

### 3.3 Skill 管理

- 自动发现安装包内置 Skills、Skill 市场 Skills 和用户导入 Skills。
- 用户可选择一个包含 `SKILL.md` 的文件夹，由应用复制到受管目录。
- 导入的 Skill 只进入本地 Skill 市场，默认不授权给任何机器人。
- 用户可导入包含 `app.json` 的自定义应用目录；导入后进入能力目录，默认不授权给任何机器人。
- 应用必须内置可学习的自定义应用模板，至少覆盖可执行 Node 应用和需要 Owner 审批的可执行应用示例；模板应在能力页标记为内置模板，不能被用户卸载。
- 能力页应提供自定义应用 manifest 编辑入口：本地导入应用可编辑并保存 `app.json`，保存前必须校验并在失败时回滚；内置模板必须先复制为本地副本后才能编辑。
- 用户可升级或卸载已导入的自定义应用；卸载前必须确认没有 Bot 授权引用或套件依赖。
- 应用必须内置可学习的套件模板，至少覆盖一个 Workflow 组合场景和一个定时任务场景；模板应在能力页标记为内置模板，不能被用户卸载。
- 能力页应提供套件 manifest 编辑入口：本地导入套件可编辑并保存 `suite.json`，保存前必须校验 JSON、Workflow 步骤和阻断诊断，并在失败时回滚；内置模板必须先复制为本地副本后才能编辑。
- 用户可升级或卸载已导入的套件；能力页必须展示套件版本、发布者、可信来源、标签、manifest 诊断和本机生命周期，卸载前必须确认没有 Bot 授权、命令或定时任务引用。
- Bot 可挂载统一能力引用。旧版 Skill 授权继续兼容，但自定义应用、后续套件、MCP、命令和定时任务都必须通过 Bot 维度的能力引用治理。
- 自定义应用必须声明入口、调用面和权限需求；授权前应展示这些信息、manifest 诊断和生命周期状态。
- 自定义应用可声明 `desktopAutomation` 权限元数据，用于探索 macOS 桌面辅助能力。声明屏幕录制、辅助功能、剪贴板或键盘输入时必须在能力治理中标记高风险；当前 PoC 只允许可见未读读取和草稿模式，`autoSend=true` 必须阻断导入或升级。
- 微信桌面辅助能力不得作为微信协议机器人或 IM Provider 宣称；当前只作为自定义应用 PoC，最多尝试截取微信当前窗口，并通过本地初筛和已配置多模态模型识别可见未读。识别到多个可见未读候选时可按队列读取有限数量会话；同一会话多条消息只读取当前窗口可见内容，不承诺读取不可见历史。模板可选把回复草稿写入系统剪贴板，用户需要手动确认会话、粘贴和发送。
- 自定义应用返回内容应支持按应用配置后处理：每个自定义应用默认原样返回；用户可为某个应用单独选择让主进程使用当前模型配置进行一次无工具文本总结后再回复。模型 API Key 不得下放给自定义应用脚本。
- 自定义应用输出协议应支持受控结果投递请求。应用只能按 `routeId` 请求投递到当前 Bot 已启用的 `deliveryRoutes`，不得自行指定任意 `chat_id`、平台凭据或发送身份；主进程必须在回复后处理完成后统一校验、发送和记录结果。
- 自定义应用预留入口必须有友好 UI 响应：`webview`、`mcp-adapter` 和 `workflow` 当前只展示建设中说明，不得出现在命令或定时任务可执行目标中。
- 能力页应提供扩展治理诊断，至少覆盖自定义应用入口和权限风险、套件缺失依赖、Workflow 步骤引用缺失，并给出授权前处理建议。
- 能力页应提供 Bot 维度治理控制台，按 Bot 展示授权引用、policy、命令/定时绑定和最近能力使用审计。
- 能力页应提供 IM 和连接器诊断，按 Bot 展示主消息平台、企业微信封闭状态、飞书知识连接器和结果投递路由的配置缺口。
- 命令、定时任务和 Owner 审批阻断应写入 Bot 维度能力使用审计，审计记录不得作为授权来源。
- Bot 支持配置和新增 `/xxx` 命令映射和别名；当前命令目标至少支持已授权 Skill、已授权 MCP、已授权套件、已授权套件下派生的 Workflow，以及已授权且声明 `commandCallable` 的自定义应用。
- Bot 授权能力应支持配置运行策略，至少覆盖 Agent、命令、定时任务和 Owner 审批四类治理开关。
- 命令配置 UI 应提示保留命令和命令名/别名冲突，避免保存后运行时不可用。
- 系统保留命令 `/new`、`/continue`、`/owner`、`/help` 不得被用户配置覆盖；`/help` 应自动列出当前 Bot 已启用命令、别名和说明。
- 能力引用或自定义应用声明需要 Owner 审批时，运行时不得直接执行该能力；命令调用应创建 Owner 审批请求，定时任务应失败并记录需要审批的原因。
- Workflow 声明式步骤应支持步骤级输入模板、命名步骤输出变量、条件跳过、循环、失败恢复、超时和重试，执行日志需要展示跳过状态和重试尝试次数，避免单个扩展能力长时间阻塞整条流程。
- 定时任务当前至少支持 `interval`、`daily`、`weekly` 和 5 段 `cron` 四种计划类型，目标至少支持 `agent`、已启用命令、已授权 Skill、已授权 MCP、已授权套件、已授权套件下派生的 Workflow，以及已授权且声明 `scheduledCallable` 的自定义应用。
- 支持配置全局 MCP 服务；当前可保存 `stdio`、HTTP 和 SSE，其中只有 `stdio` 在 Bot 维度显式授权后注入 Claude Agent SDK，HTTP / SSE 作为后续运行时接入的占位配置。
- Agent 默认注入内置 Playwright MCP 工具，无需用户在 MCP 配置页单独添加。该工具提供网页打开、点击、输入、页面结构读取、网络请求查看和截图验证能力；运行时应使用隔离的 headless 浏览器会话，不复用用户日常浏览器登录态，产物写入当前会话 workspace。
- GUI 应展示 MCP 配置诊断，至少覆盖启用状态、传输类型、命令解析或 URL 缺失、cwd 可读性、环境变量缺值、Bot 授权情况、stdio 协议握手和工具列表预览。
- Bot 挂载的套件应进入当前 Agent 上下文，向模型提供该套件声明的说明、工作流以及其下已获授权的 Skill、自定义应用和 MCP 摘要，供模型自主决策使用。
- 本地技能市场展示 Skill 名称、描述和来源，并允许删除用户导入的 Skill；删除时同步撤销对应机器人授权。
- 用户可配置 HTTPS Git 仓库、分支和可选 Token 作为 Skill 市场。
- 用户 Skill 在同名冲突时优先于市场和内置 Skill。
- 用户可按机器人勾选允许访问的 Skills；未授权 Skill 不可见、不可调用。
- Skill 较多时，机器人授权界面支持按名称或描述搜索，并对当前筛选结果批量授权或取消。
- 本地技能市场支持按来源和未授权状态筛选，并展示 Skill 已授权给哪些机器人。
- 点击技能市场中的 Skill 可预览 `SKILL.md` 和文件清单；导入同名或同目录 Skill 时必须提供冲突解决，支持以新的为准覆盖、以旧的为准保留现有版本，或打开对应 `SKILL.md` 让用户手动编辑后再决定，不得静默隐藏或自动吞掉冲突。
- 用户可从 Skill、自定义应用和套件卡片打开资源所在目录，便于复制、检查或备份资源文件；渲染层不得传入任意路径，只能请求主进程打开已发现资源目录。
- 本地多个 Skill 的 frontmatter `name` 相同时，后续目录必须仍可在技能市场中以目录名区分显示。
- 正在被任一机器人授权使用的本地 Skill 不允许删除。
- 不支持全量通配授权；新增 Skill 必须由用户显式勾选后才能被机器人使用。
- Skill 可维护自身 `knowledge/`、`references/` 和 `scripts/`。

### 3.4 消息、会话与多模态

- 支持飞书私聊和群聊艾特消息；企业微信私聊/群聊消息事件适配代码暂时保留但入口封闭，不进入当前验收范围。
- 私聊按 chat 保持连续上下文；群聊按 chat 与发送者组合隔离上下文。
- 会话无活动 24 小时后过期。
- 每个机器人可配置长任务自动提示：当单次消息处理超过指定秒数仍未完成时，自动向原会话先回复一段可配置文案；最终结果完成后仍正常回复。
- 用户可发送 `/new`、`新对话` 或 `重置会话` 主动创建新上下文。
- 图片消息自动下载并作为多模态输入。
- Word、PowerPoint、Excel 附件自动预处理；PowerPoint 在多模态开启时附带视觉预览。
- 找到高度匹配但下载耗时的飞书文件时，可先给出基本回答并创建待确认任务；用户通过 `/continue <任务编号>` 确认后继续下载和分析。
- Agent 需要下载飞书云盘文件或导出云文档继续分析时，应通过应用受控 helper 触发，优先命中应用级文件缓存；helper 只把文件物化到当前 Bot 当前会话 workspace，不暴露全局缓存目录。
- Agent 直接使用 Bash 裸调 `lark-cli drive +download` 或 `drive +export` 应被运行时识别并拦截，引导走受控 helper。

### 3.5 存储管理

- 展示可清理会话的占用、机器人、对话标识、更新时间和过期状态。
- 存储管理必须区分会话数据和文件缓存；用户可单独清理应用级内容哈希缓存。
- 存储管理必须单独展示自定义应用运行产物，例如会话 workspace 下的 `apps/<app-id>/` 截图、临时 JSON 和调试文件；用户可清理全部或过期产物，且不得删除自定义应用本体、Bot 配置、授权、飞书 OAuth 或用户 Skills。
- 用户可配置自定义应用运行产物保留天数和是否自动清理过期产物；自动清理只能作用于明确过期的运行产物，不得清理正在使用的配置或应用安装目录。
- 存储管理应展示应用级文件缓存索引，至少包含来源类型、关联 Bot、文件名、大小和来源摘要，并支持按 Bot 和来源类型筛选以及单条缓存删除。
- 存储管理应展示定时任务最近运行历史，但会话清理和文件缓存清理不得删除该审计记录。
- 存储管理应展示历史补处理 Beta 游标占用，并提供单独清理入口；会话清理不得误删游标，避免用户清理会话后失去断网补处理起点。用户主动清理游标后，补处理需要等待重新收到消息建立新起点。
- 文件缓存应支持按索引单条删除和保守自动失效；自动失效只能清理有明确缓存时间且超过保留周期的索引，不能误删仍被其他索引引用的共享内容 hash。
- 文件缓存索引应展示新鲜度状态；缺少 freshness key 的远端文件缓存必须标记为新鲜度未知。
- 文件缓存应提供索引校验和修复入口，用于移除缺失文件索引、清理孤立内容 hash，并补齐可安全恢复的索引字段。
- 用户可点击会话查看 Claude 会话 ID、关联消息、最近对话记录和 workspace 文件清单。
- 会话详情应提供便于阅读的消息明细视图，至少区分接收消息、Agent 可观察工作过程和最终回复内容；详情弹窗应支持按事件类型筛选，并允许导出当前会话详情 JSON。
- 支持清理过期会话、勾选会话或全部会话。
- 会话清理不得删除应用配置、飞书授权和用户 Skills。

### 3.6 UI 与可观测性

- GUI 支持配置、监听启停、运行状态、日志、Skill 列表、Skill 市场和存储清理。
- GUI 的主要功能页应使用页内标签或等价分组导航组织内容；运行台、配置、Skill 市场、能力治理、存储管理和定时任务中心都不得把所有配置和记录堆在一个长页面里。
- 新增或编辑复杂配置应优先使用弹窗、详情弹窗或局部编辑器承载，例如 Bot、定时任务、自定义应用详情和 Manifest，避免在主页面直接展开成长表单。
- GUI 提供能力治理页，展示统一能力目录和已导入自定义应用；能力页信息必须分层组织，至少把治理总览、诊断排障、MCP 服务、套件/Workflow、自定义应用和使用审计拆成可切换子页面，避免所有能力堆叠在同一长页面。
- GUI 支持应用级主题切换，至少提供跟随系统、浅色和深色三种模式。
- macOS 应用图标必须使用固定的规范 icon 资源，不能因为界面 Logo 或品牌区样式调整而被隐式替换。
- GUI 左上角 Logo 提供应用内使用手册，说明快速开始、配置项、Bot、Skill 市场、运行台、飞书权限、能力治理和存储管理，并提供飞书问答、企业微信暂时封闭说明、命令、定时任务、套件/Workflow、MCP 和升级恢复等场景化教程。
- GUI 中关键配置项应提供 `?` 说明入口，说明该项是什么、何时需要填写、缺失或错误时会造成什么影响。
- GUI 全局状态应按多 Bot 聚合展示在线数量、监听数量和排队任务数，不使用单一 RUNNING/STOPPED 表示全部机器人。
- 机器人配置在配置页以列表展示，点击后通过弹窗编辑和保存详情；配置项应提供可点击说明入口。
- 机器人编辑弹窗应按基础、平台、投递、Skills、能力、命令和定时任务分组展示，提供左侧或顶部标签导航，避免所有配置堆叠在单个长表单里。
- 长内容弹窗应固定标题或操作区并让主体内容独立滚动；发布说明、使用手册、帮助说明、预览/Manifest、自定义应用详情、会话详情、运行详情、定时任务编辑和 Bot 编辑等弹窗不得因为内容过长导致底部配置不可见。
- 机器人编辑弹窗应根据消息平台动态展示配置项：飞书展示飞书接收/回复身份、处理中表情、Owner 和 OAuth；企业微信展示“暂时封闭”反馈并禁用缓存初始化、聊天列表、事件桥和轮询配置；钉钉展示建设中反馈，不得把不同 Provider 的专属配置混在同一个无差别表单中。
- GUI 展示当前应用版本；点击版本号后显示面向用户的更新记录。
- 点击运行台中的机器人后展示其独立详细日志，并支持按日志等级筛选。
- 执行日志页应提供“一键导出排障包”能力，让用户选择保存 ZIP 文件后发给支持排查。排障包至少包含 README、脱敏配置/快照、内存日志、落盘日志、存储统计、定时任务历史、MCP/平台诊断和能力审计摘要；必须脱敏已知密钥、Secret、Token、密码等字段，并提示日志仍可能包含用户消息正文、文件名或 chat id。
- 点击启动机器人监听后必须立即给出可见运行日志；启动前置阶段失败或超时不得静默无反馈。
- 运行台日志默认记录 Agent 可观察工作过程；是否向飞书用户发送进度消息由 Bot 配置单独控制。
- 日志记录消息接收、处理结果、错误、飞书事件投递延迟，以及资源处理、Agent、飞书回复和总耗时。
- 停止后再次启动监听必须可正常工作，不得残留重复监听状态。
- 应用启动时必须立即请求远端授权文本，运行中应按随机间隔复检同一授权地址。内容明确为 `Auth=open` 时继续运行，并重置网络不可达累计；内容明确为 `Auth=close`、格式异常或 HTTP 异常时停止 Runtime 并退出。网络不可达只累计不立即关闭：从第一次不可达开始，连续 90 天以上且累计 200 次以上检测都不可达时，才视为未授权并退出。

## 4. 非功能需求

- 凭据和用户数据仅保存在本机，不进入 Git。
- macOS Agent sandbox 允许调用系统 trustd 完成内置 Go CLI 的代理 TLS 校验，但仍禁止 unsandboxed 命令并保持目录隔离。
- 单个对话内任务串行，跨对话并发受配置限制。
- 超出跨对话并发上限的消息进入处理队列，并在运行台展示排队数量。
- 单次 Agent 最大步数可配置，默认 60；复杂 Skill 或飞书资料检索不得被固定 20 步上限过早中断。
- 应用重启后可恢复未过期会话。
- Office 压缩文件解析必须限制条目数量和解压体积。
- 发布包包含运行所需的 Electron、Claude 运行时、飞书 CLI、企业微信 CLI、Git 和 Office 解析能力。
- 飞书 CLI 必须支持本机升级路径：运行时优先使用 Bot 显式配置的 CLI 路径，其次检测本机已安装的可信 `lark-cli`，最后才使用安装包内嵌 CLI；未安装本机 CLI 的普通用户仍应拿来可用。
- 内置 Playwright MCP 当前采用轻量 Chrome channel 运行方式；若后续要求完全不依赖目标机器已安装浏览器，发布包需要额外捆绑 Playwright Chromium 并在打包验证中核对体积与可执行路径。

## 5. 当前非目标

- 不支持 Windows 或 Linux 桌面端。
- 不支持 SSH Skill 市场地址。
- 不提供云端集中式机器人管理或用户数据同步。
- 不保证所有 PowerPoint 元素都能被 Quick Look 完整渲染。

## 6. 核心验收标准

1. 在未安装开发环境的新 macOS 上，配置模型和 IM 机器人后可以启动监听并回复消息。
2. 停止监听后再次启动成功，不出现已有监听的残留错误。
3. 两个机器人只能访问各自授权的 Skills 和会话数据。
4. 同一私聊在 24 小时内保持上下文；群聊中不同发送者互不共享上下文。
5. 收到消息后可见处理中表情，处理完成后表情被移除并收到最终答复。
6. Word、PPT、Excel 和图片消息可被处理，PPT 在多模态开启时使用视觉预览。
7. 选择性清理只删除目标会话，不影响配置、授权和 Skills。
